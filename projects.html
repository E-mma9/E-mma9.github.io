Monitoring Homelab
Completed 3-node Proxmox VE cluster
View on GitHub

Full enterprise-grade monitoring stack on a 3-node Proxmox VE cluster with consumer hardware. All configuration written manually from the official documentation — no install wizards, no pre-built templates. The goal: understand and be able to explain every single line.

The stack scrapes metrics from all nodes and services every 15 seconds, stores them for 180 days in Prometheus, visualises them live in Grafana and sends an SMS via PagerDuty on incidents — and automatically calls your phone if you don't respond.

Proxmox VE Prometheus Grafana Alertmanager PagerDuty node_exporter blackbox_exporter Tailscale Linux systemd

Components
Prometheus
port :9090 — metrics store & evaluator

The core of the stack. Pull-based: scrapes metrics from all targets every 15s via HTTP. Stores time-series data locally and continuously evaluates alert rules.

PromQL for dashboard queries and alert expressions
180 days retention — trends and historical incidents
Self-monitoring: also scrapes itself (:9090)
Grafana
port :3000 — visualisation

Connects Prometheus as a data source and displays all metrics as real-time graphs, gauges and tables. Stores no data itself — always live queries.

Cluster overview: CPU, RAM, disk per node side-by-side
Alert status panels — immediate visual feedback
Accessible via Tailscale VPN from anywhere
Alertmanager
port :9093 — alert routing

Receives fired alerts from Prometheus, groups duplicates and delivers them to PagerDuty. Separation of detection (Prometheus) and delivery (Alertmanager).

group_by alertname — 3 nodes down = 1 incident
30s group_wait — bundles related alerts together
send_resolved: true — automatic resolved notification
Blackbox Exporter
port :9115 — external probing

Performs real HTTP/HTTPS and ICMP checks on all services from the outside — catches problems before users notice them.

6 services: Vaultwarden, Grafana, Nginx PM, Pi-hole, Paperless, Glance
ICMP ping checks on all 3 Proxmox nodes
SlowResponse alert when response time >5 seconds

Config Spotlight
/etc/prometheus/prometheus.yml
global:
scrape_interval: 15s # industry standard — fast enough, not too heavy
evaluation_interval: 15s
 
alerting:
alertmanagers:
- static_configs: [{targets: ["localhost:9093"]}]
 
scrape_configs:
- job_name: 'proxmox-nodes'
static_configs:
- targets:
- '192.168.1.249:9100' # pve — primary + ZFS storage
- '192.168.1.211:9100' # node2 — services node
- '192.168.1.74:9100' # node3
labels: {group: 'proxmox'} # PromQL: up{group="proxmox"}
 
- job_name: 'blackbox-http' # HTTP checks: expects 200/301/302
metrics_path: /probe
params: {module: [http_2xx]}
# relabel_configs routes request through blackbox_exporter
/etc/prometheus/alert_rules.yml
# Pending → Firing → Alertmanager → PagerDuty → your phone
groups:
- name: node_alerts
rules:
 
- alert: NodeDown
expr: up{job="proxmox-nodes"} == 0
for: 1m # 1 min wait filters short network blips
labels: {severity: critical}
 
- alert: DiskSpaceWarning
expr: 100 - (node_filesystem_avail_bytes{mountpoint="/"}
/ node_filesystem_size_bytes{mountpoint="/"} * 100) > 85
for: 5m # filters temporary spikes from large file copies
 
- alert: HighMemoryUsage
expr: 100 * (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) > 90
for: 5m
 
- name: blackbox_alerts
- alert: EndpointDown
expr: probe_success == 0
for: 2m # 2 min — room for container restarts
/etc/alertmanager/alertmanager.yml
global:
resolve_timeout: 1m # anti-flapping: wait before sending "resolved"
 
route:
receiver: 'pagerduty'
group_by: ['alertname'] # 3 nodes down = 1 incident, not 3
group_wait: 30s # groups related alerts together
group_interval: 5m
repeat_interval:4h # reminder if issue persists
 
receivers:
- name: 'pagerduty'
pagerduty_configs:
- severity: '{{ if eq .CommonLabels.severity "critical" }}critical{{ else }}warning{{ end }}'
description: '{{ range .Alerts }}{{ .Annotations.description }}{{ end }}'
send_resolved: true # notification when issue is resolved
What I learned
Proactive vs reactive
With the right thresholds and for-durations you catch problems before users notice them. Alert-tuning is a skill — too sensitive and you go blind to noise.
Writing config from scratch
Writing everything from the documentation without templates teaches you what every field does. Relabeling in Prometheus seemed like magic — now I understand every step.
Separation of concerns
Prometheus detects, Alertmanager delivers. That separation makes debugging and extending much simpler — changing one component doesn't affect the rest.
Linux Security Hardening
Completed Ubuntu / Debian automation
View on GitHub

Two Bash scripts that automatically harden and audit an Ubuntu/Debian server. Written to learn Linux security fundamentals hands-on — from SSH lockdown to firewall configuration and audit logging. Tested on Ubuntu 22.04 LTS.

harden-server.sh configures the system in a single run: updates, SSH hardening, UFW firewall, auditd installation. security-audit.sh then reports the current security status — SSH config, firewall rules, failed login attempts and SUID files.

Bash Linux SSH UFW auditd Ubuntu 22.04 Debian
What harden-server.sh does
1
System updates
Runs apt upgrade and enables unattended-upgrades for automatic security patches — no forgetting to update manually.
2
SSH hardening
Disable root login, turn off password auth, change port to 2222 — only key-based authentication allowed.
3
UFW firewall
Default deny incoming. Only SSH (2222), HTTP (80) and HTTPS (443) explicitly allowed. Everything else blocked.
4
auditd installatie
Installs and enables auditd for security event logging — audit trail of system calls, file accesses and user actions.
5
Automatische security audit
Runs security-audit.sh immediately after hardening — checks SSH config, firewall status, SUID files and failed SSH logins. Gives instant visual confirmation that everything is correctly configured.
